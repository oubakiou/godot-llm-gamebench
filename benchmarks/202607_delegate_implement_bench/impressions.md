[^score]: 100 点満点のルーブリック（隠しテスト 60 / 決定性 10 / 型品質 15 / プロジェクト健全性 15。[delegate_implement_bench_design.md](../docs/design/delegate_implement_bench_design.md) の「採点」）による全項目自動採点の、採用（completed）ラン合計（各モデル 3 反復、最大 300。反復間の再現性を評価へ含めるため中央値でなく合算）。型品質は strict 警告（untyped/unsafe）を error 昇格した per-file 検査のカウントで、警告 1 件につき −3（下限 0）。completed が 3 ラン未満のモデルは不足分を 0 とみなさず n を明記する（例: claude-haiku-4-5 は n=1。信頼性は本文の完了/試行で別軸報告）。

[^time]: completed ランの壁時計時間の中央値。子の実装時間だけでなく、親の委譲準備・受け入れ検証・最終報告まで含むラン全体の所要時間（親のオーバーヘッドは全モデル共通条件）。

[^quality]: LLM ワーカーによる定性評価（主判定 = claude-sonnet-5、クロスチェック判定 = gpt-5.6-sol、乖離時の裁定 = 運用者 Fable。規則は後述）。自動採点の型警告カウントが「注釈の量」を測るのに対し、こちらは型付けの徹底度・Model/View 分離・可読性・定数化といった設計面を測る補完軸（例: `Array[int]` で全部に型を付ければ警告 0 になるが enum 型を捨てた設計は本欄で減点される）。**全採用ラン（各モデル 3 ラン、claude-haiku-4-5 のみ completed 1 ラン）を共通の較正アンカーで精査**し（追試 1・2 と sol 再精査はそれぞれ各欄記載のアンカーセットを使用）、ランごとに 5 段階で採点、各判定者の値はその 3 ラン平均（小数第 1 位）。列の先頭は下記の規則による代表値 + コメント。5 段階の定義: **5** = typed コレクションを内部実装まで貫徹し magic number ほぼなし / **4** = 公開 API は typed だが内部に無型コレクションや直書き数値が残る / **3** = enum・class を宣言しつつ内部表現で型を放棄（int 運用・生 Dictionary 等）/ **2** = 無型 var が広範で構造も脆い / **1** = 型注釈がほぼ皆無（該当なし）。中間評価は 0.5 刻み。各モデル欄の ◎/○/△/▲ 表記はそれぞれ 5/4/3/2 に対応する。ばらつきの詳細は各モデル欄を参照。列の表記は「代表値 (sonnet評, sol評)」。代表値は、(1) 判定者間乖離が誤差帯（±0.5）を超え Fable 裁定がある行は裁定値（claude-sonnet-5 = 3.8、gpt-5.4-mini = 3.3、cursor-gemini-3.1-pro = 2.5、追試 1 の gpt-5.6-luna@xhigh = 3.8）、(2) 裁定がなく判定が割れた行は両判定の平均を小数第 1 位で切り捨てた値、(3) それ以外は sonnet 評。gpt-5.6-luna の sol 評はアンカー使用のため「—」、cursor-gemini-3.1-pro と fable-direct の sol 評は n=2。判定者間の乖離と解釈は「判定者クロスチェック」欄を参照。

非 completed の詳細（完了/試行 = 採用数/全試行数: gpt-5.5 3/3、gpt-5.4-mini 3/4、devin-glm-5.2 3/5、swe-1.6 3/3、claude-sonnet-5 3/3、claude-opus-4-8 3/6。追加ラウンド: gpt-5.3-codex-spark 3/3、devin-deepseek-v4-pro 3/4。ラウンド 3 で再計測した Cursor 系は旧ラウンド・混雑窓・追加ラウンド 2 の全試行を分母に含めて composer-2.5 3/10、composer-2.5-fast 3/10、cursor-gemini-3.1-pro 3/13、cursor-kimi-k2.7-code 3/10、cursor-grok-4.5 3/15）は各モデル欄を参照。failed 3 件（+追加ラウンド 1 件）はすべて親側要因（セッション枠 429 ×2、親のプロトコル逸脱 ×2）で、子モデルの失敗ではない。追加ラウンドの stalled 1 件も response 受領後の親の沈黙で子は無罪。ラウンド 3 の failed 10 件もすべて親側要因（session 枠 429。うち 9 件は 5h 枠 100% 時間帯に子未起動のまま数秒で失権した即時失敗）で、stalled 7 件の内訳は親の沈黙 ×2（composer 系）と子側の停滞 ×5（すべて cursor-gemini-3.1-pro。rep1 ×4 + rep0 ×1。計測の経緯のラウンド 3 参照）。claude-haiku-4-5 は 1/7（5 stalled・1 failed）で、stalled 5 件はすべて子側（自作の終了しない godot 待ち。haiku 欄と issue #7 参照）、failed 1 件は子 exit 1 に親の一語規約逸脱が重なったもの（成果物の残存採点は 85 — 機能テストは全 pass だが型警告 80 件）。

## ベンチ対象モデル（採用結果）

サマリー表と同順。各欄は採用ラン（(model, rep) ごとに最新 completed）に基づく最終所感で、コード品質の記述は sonnet-5 判定に基づく（代表値の規則と判定者間乖離は「判定者クロスチェック」欄）。再計測で置換された旧ラウンドの記録は「計測の経緯」欄に残す。

### claude-sonnet-5（Claude CLI）

本計測: **3 反復すべて completed・85 / 100 / 100（合算 285 で委譲モデル中の品質首位）、再実行ゼロ**。7.9〜9.5 分・全ラン 1 往復。型警告ゼロの完全クリーン run を 2 回出した唯一の委譲モデル（rep0 のみ警告 20 件）。なお警告 0 の rep2 が定性評では △ 側なのは「`Array[int]` で注釈は完備しつつ enum 型を捨てる」パターンで、警告カウントと設計品質が別軸であることの好例。

- 事前に懸念していた枠逼迫時の停滞は一度も発生しなかった（枠に余裕がある時間帯に実行）
- stream-json 化（delegate-skills v0.6.0）の効果で、実測コスト付きの子トークンが取れた（$1.28〜$1.39/ラン、in 3.8k〜4.1k / out 35.7k〜40.1k。Claude 実行系共通の恩恵で、後続の opus4.8・claude-haiku-4-5 でも同様に実測）。Codex 系と異なり入力にキャッシュ読みが含まれない模様で、トークン構成の直接比較には注意
- コード品質（3 ラン精査、○）: rep1 は `_grid: Array[CellKind]`・`_items: Array[_ItemState]`・`BELT_KEYS: Array[BoardModel.CellKind]` の全面型付けで委譲モデル中の最上位級（◎ 相当）だが、rep0/rep2 は `_grid: Array[int]`・`_items: Dictionary` と内部で型を放棄。単一ラン精査（rep1、◎）は上振れだった。ぶれの軸は「内部状態が enum 型を保つか int に落ちるか」。判定者クロスチェックで乖離（sol 3.3）が出たため、サマリー表の代表値は Fable 裁定 3.8（クロスチェック欄参照）

### swe-1.7（Devin CLI）

本計測: **3 反復すべて completed・85 / 100 / 88（合算 273、委譲モデルで sonnet5 に次ぐ歴代 2 位）、再実行ゼロ**。10.5〜12.8 分・全ラン 1 往復、実測 in 2.16M〜3.79M / out 51.7k〜60.0k。rep1 は型警告ゼロを含む 100 点満点（gpt-5.5・sonnet5 に続く 3 モデル目）。

- **swe-1.6 からの改善幅（+60.72）は世代間比較として今ラウンド最大**。一方で所要時間は 3.9 分 → 12.5 分と最遅級へ悪化し、入力トークンも 2〜4 M と多い。無料プレビュー（2026-08-08 まで）のため子費用 $0 で、総額 $1.41 は完走モデル中最安
- コード品質（3 ラン精査、○）: 平均 4.0 だが分散が最大（4.5 / 3.0 / 4.5）。rep0/rep2 は typed コレクション貫徹（rep0 は Dictionary を使わず typed Array のみで統一）、rep1 だけ `_items: Dictionary = {}  # id -> _Item` のコメント型注釈 + `as` キャストへ退行。型警告数（6 / 0 / 4）とは逆相関気味で、警告カウントと設計品質が別軸であることの新しい例

### gpt-5.5（Codex CLI）

本計測: **3 反復すべて completed・85 / 100 / 85（合算 270、委譲モデル 2 位）**。全 rep 1 往復、5〜6 分、実測トークン in 402k〜566k / out 9.8k〜13.0k。機能面は隠しテスト 33 ケース全 pass で反復間のぶれが皆無、仕様の細部（スプリッタ反転・RNG 非消費）を毎回落とさない。減点は型警告のみ（rep0 25 件 / rep1 0 件 / rep2 5 件）で、型品質の再現性が課題。

コード品質（3 ラン精査、○）: rep2 のみ `_items: Array[_ItemState]` まで typed 徹底の ◎ 相当だが、rep0/rep1 は中核の `_items: Dictionary` が無型。3 フェーズ分離と `BoardModelScript` 経由のカプセル化は 3 ラン共通で綺麗。単一ラン精査時の ◎ は rep 選びの上振れだった。

準備フェーズでの実績（ベンチ外）:

- リファレンス実装（2026-07-03、約 8 分、1 往復で completed）。不動点反復による移動解決、スプリッタのトグル反転条件、RNG 消費規則、`peek_next_kind()` の非消費実装（state 複製）など、仕様書の細部を一切落とさず、親レビューでの指摘事項ゼロ
- 検証が自発的: 指示した検証 3 件に加え、追加のルール検証スクリプトを自作して実行。報告の Verification 記載も正確（親の再実行で全件再現）
- Typed GDScript の品質が高い（内部クラス・typed Array を適切に使用。改善余地は typed Dictionary 未使用程度）。報告プロトコルも正しく遵守し読み取りコストが低い
- 隠しテストランナー（33 ケース、662 行、約 9 分、1 往復。sonnet 停滞打ち切り後の再委譲）。テスト設計・堅牢化は的確だったが、seed の偶然で pass する期待値の off-by-one が 1 件混入（親レビューの実測裏取りで検出・修正）。「テストが通った」報告の裏に潜在バグが残り得る好例で、オラクル側成果物は親の独立検証が必須という運用知見

- rep1 初回は**親（Fable）の最終応答プロトコル逸脱で failed**（受け入れ基準の説明文を付けてから completed と書いた。子は無罪、残存 88）→ 撃ち直しで 94

## 追試（2026-07-11、gpt-5.6-luna の reasoning effort A/B: medium vs xhigh）

追加ラウンド 2 で gpt-5.6-luna が「スプリッターの反転仕様を系統的に落とす」観察を得たことを受け、reasoning effort で結果が変わるかを同一ベンチで A/B 計測した。Codex CLI 0.144.1 は `-c model_reasoning_effort=<値>` で effort を指定でき、gpt-5.6-luna のデフォルトは medium（＝追加ラウンド 2 のランは実質 medium 条件）。ラウンド間変動と交絡させないため過去ランを流用せず、**medium / xhigh の両アームを同日に交互実行で各 3 反復**、新規に計測した。CLI 固定バージョンはラウンド 2/3 と同一。集計キーは `gpt-5.6-luna@medium` / `@xhigh` として本計測の行とは分離している（effort 指定条件のため本計測サマリー表とは直接比較せず本欄で読む）。

| 条件                | 自動テストによる評価(合算)[^score] | コード品質 (sonnet評, sol評)[^quality]                                       | 親費用+子費用(中央値)[^cost]    | 所要時間(中央値)[^time] | ひとこと                                       |
| ------------------- | ---------------------------------: | ---------------------------------------------------------------------------- | ------------------------------- | ----------------------: | ---------------------------------------------- |
| gpt-5.6-luna@medium |                             244.09 | 3.7 (3.7, 3.7) — rep 間振れは相変わらず大（3.0/3.5/4.5）                     | $1.59 + $0.71（計 $2.30、換算） |                  8.0 分 | デフォルト effort。ラウンド 2 を再現           |
| gpt-5.6-luna@xhigh  |                             285.00 | 3.8 (3.5, 4.2) — 警告ゼロ ×2 でも設計面は実質据え置き（代表値は Fable 裁定） | $1.34 + $1.75（計 $3.09、換算） |                 13.0 分 | 機能満点 3/3・100 点ラン ×2、往復は全ラン 1 回 |

- **「系統的な弱点」は effort で消えた**: medium は今回もスプリッター系を 3 反復中 2 回失敗（rep1 は collision ×2・spawn_miss も、rep2 は win_path のみ失敗）とラウンド 2 の観察を再現。xhigh は 3 反復すべて隠しテスト全 pass。仕様細部の読み落としが「モデルの癖」ではなく「思考予算の不足」だったという切り分け

- 代表値の方針（2026-07-12 のユーザー判断）: (1) 乖離が誤差帯（±0.5）を超え Fable 裁定がある行は裁定値（本計測の claude-sonnet-5 = 3.8 / gpt-5.4-mini = 3.3 / cursor-gemini-3.1-pro = 2.5、追試 1 の gpt-5.6-luna@xhigh = 3.8）、(2) 裁定がなく判定が割れた行は両判定の平均（小数第 1 位切り捨て）、(3) sol 評が無い行は sonnet 評単独。各モデル欄の記述は sonnet-5 判定に基づくため、序列に関わる結論を引用する際は本欄の両判定値を併記することを推奨

## ラウンド横断の観察

- **天井効果は機能軸に限った現象**: 隠しテスト（60 点軸）は前ラウンドの委譲 7 モデル中 4 モデル（+ベースライン fable-direct）が全反復満点と天井気味だが、型品質軸が差を広げ、合算は 212.28〜285.00 に分布。型警告ゼロの run を出せたのは gpt-5.5（1 回）と claude-sonnet-5（2 回）だけで、fable-direct（3 回）を除けばゼロ警告はむしろ例外。機能・型の両軸で崩れたのは swe-1.6 のみ
- **「実装完成後・完了報告前」の停滞は実行系横断の現象**: Codex（gpt-5.4-mini）・Devin（devin-glm-5.2）・Claude（claude-opus-4-8）の 3 実行系で各 1 回観測。特定ベンダー固有ではない
- **親側交絡が失敗の主因**: failed 3 件はすべて親（claude -p）側の要因（セッション枠 429 ×2、最終応答のプロトコル逸脱 ×1）。子モデルの評価には影響させず、信頼性メトリクスの解釈時に注記する
- **委譲プロトコルの時間コストは約 3 分/ラン**: 同一実行系（Claude）で委譲あり（sonnet5 / opus4.8: 8.4〜8.6 分）と委譲なし（fable-direct: 5.7 分）を比較すると、往復 1 回の委譲でも準備・観測・報告読取りで 3 分前後を消費している。費用は総額ではほぼ同等
- **型警告カウント（自動）と定性評は補完関係**: 警告数は「注釈の量」を測り、定性評は「設計」を測る。gemini（警告 9〜82 件・定性 ▲〜△）のように両者が符合する例もあれば、deepseek（警告 4〜5 件と少ないのに内部は生 Dictionary 運用で △）や sonnet rep2（警告 0 なのに enum を int に落として △ 側）のように、注釈が完備でも設計の型崩れは警告に出ない例もある。精査すると「enum を宣言しつつ内部を int / 無型 Array で運用する」型崩れから「typed コレクションを末端まで貫徹」まで幅があり、**モデル間の差よりモデル内のラン間ばらつきが支配的**。◎ 水準を安定して出せたのはベースラインの fable-direct のみで、型品質の「安定性」自体がモデル特性として報告価値がある

## ハーネス側の注意（モデル所感ではないが委譲品質に影響）

- **Claude 実行系の子は停滞リスクが恒常的にある**: sonnet（4.6）で 2 回、haiku の E2E rep3 で 1 回、さらに claude-haiku-4-5 の正式ランで 5 回観測（正式ラン分の機構は上記「自作のハングするサブプロセス待ち」で特定済み。E2E rep3 はレート制限解除直後にもかかわらず、実装の大半を書いた後 17 分以上無出力で停滞 → watchdog の 40 分絶対上限で timeout 打ち切り）。timeout 打ち切り→残存成果物の採点→metrics 記録の経路は rep3 で実地検証済み。なお SIGKILL 打ち切りでは親の usage が取得できず parent_tokens は null になる
- **Claude 実行系は枠逼迫時に停滞・沈黙しやすい**（2026-07-03〜04 の sonnet 委譲で観測）: 初回は response 未生成のまま exit 1（stderr は /tmp 掃除で消失し原因確定できず）、リトライはプロセス生存・CPU 微増のままファイルを一切書かず 37 分無進捗で手動 kill。時期的に週間制限のスロットリングが疑わしい。レート/枠エラーでも沈黙して exit 1 になるため切り分けコストが高く、watchdog（無進捗タイムアウト）が必須
- **orchestrator が落ちても子プロセスツリーは生き残る**: watchdog crash 時（opus rep1）に親 claude -p と子ワーカーが孤児化し、枠を消費し続けていた。ハーネス障害後は必ずプロセス残骸を確認・停止する
- シェル環境に `DELEGATE_IMPLEMENT_MODEL` が残留することがある。委譲時は毎回明示指定しないと意図しないモデルに解決される（sonnet 指定のつもりが gpt-5.5 に解決された事例あり。v0.6.0 からは prepare 出力の `model_source` で env 由来を検出できる）
- Cursor の GLM は `glm-5.2-high` / `glm-5.2-max` のみで、素の `glm-5.2` は存在しない
- **Cursor `agent` CLI は複数プロセス同時起動で `~/.cursor/cli-config.json` の書き込みが競合して落ちることがある**（`ENOENT: rename cli-config.json.tmp`。2026-07-05 のスモーク確認を 3 プロセス並列で行った際に 1 件発生、単独再試行で成功）。ベンチ本計測は直列実行のため影響しないが、スモーク確認も含め agent CLI は並列起動しないこと。skill 側の緩和策（config 隔離 or 直列化）は [issue #8](https://github.com/oubakiou/delegate-skills/issues/8) として上流へ提案済み
- **子ワーカーが自作のハングするサブプロセスを待って停滞する**: 終了処理のない godot スクリプトを Bash 実行し、返らないコマンドを待ち続ける形。claude-haiku-4.5 で 7 試行中 5 回観測（sonnet5 0/3、opus4.8 1/6 と頻度はモデル依存）。非対話の子はツールエラーを受け取る機会がないまま watchdog SIGKILL に至るため、delegate-skills へ Bash timeout 注入等の復帰策を提案（[issue #7](https://github.com/oubakiou/delegate-skills/issues/7)）
- **停滞ランの残存実装が無限出力する場合、grader が stdout の文字列化で落ちる**（`ERR_STRING_TOO_LONG`、数百 MB 到達）: 出力捕捉に上限（先頭・末尾 4MB ずつ保全、中間切り捨て）を入れて修正済み。打ち切りラン採点はこの種の暴走成果物を前提に組む必要がある
- **親が dispatch 前に request を作り直すと「prepared のまま dispatch されない observe JSON」が残る**（2026-07-05 の gpt-5.3-codex-spark ランで 2 回観測）: 旧 metrics 集計はこれを往復数の分母に数えて usage 全数チェックが誤発動し、実測が取れているのに estimated（ほぼゼロ）へフォールバックしていた。dispatch されていない observe を分母から除外する修正を実施し、影響 2 ランの metrics は保存アーティファクトから再計算した。上流へは [issue #6](https://github.com/oubakiou/delegate-skills/issues/6) として報告済み
- **delegate-skills の `superseded` マーキング導入（issue #6 実装）に利用側の usage 全数チェックが未追随だと、実測が揃っていても estimated へ退行する**: 2026-07-10 ラウンドで発生。親が dispatch 前に request を作り直した際の旧 observe が `phase: "superseded"`（usage null、子プロセスなし）になり、`prepared` しか除外していなかった metrics.ts の分母に入って全数チェックが誤発動した。`superseded` も除外する修正を実施し、影響 12 ランを再計算済み。上流へは [issue #6 へのコメント](https://github.com/oubakiou/delegate-skills/issues/6#issuecomment-4931795333)で消費者向け注意の文書化を提案
- **cursor-agent 2026.07.09 は `create-chat` が racy に起動途中で停止する**（[issue #14](https://github.com/oubakiou/delegate-skills/issues/14)）: statsig 初期化直後・認証ユーザー解決前に無応答となり、delegate-cursor.sh の command substitution が無限ブロックして子が一度も起動されない。stdin の `/dev/null` 固定では不十分（上流が実装した `</dev/null` 版でも 5 ラン中 2 停止を確認）で、timeout 45 秒 + 最大 3 リトライの併用で解消（当初はローカルパッチ、2026-07-10 中に**上流実装済み・導入済み**でパッチ運用は終了）。watchdog が親ツリーを SIGKILL しても停止した create-chat は孤児として無期限に残る（今回 4 体を手動 kill）ため、Cursor 系の停滞ラン後はプロセス残骸の確認が必要。なお `-p` モード自体も応答完了まで stdout 無出力のため、「無言で作業中」と「本当に停止」を外部から区別できない点は引き続き注意
- **親 CLI が成功 JSON（`subtype: "success"`・`is_error: false`・一語 completed）を stdout に出しながら非ゼロ exit する事象**（2026-07-11 の追試 2、claude-sonnet-5@gq rep2 で 1 回観測）: stderr は空で、orchestrator は exit code 基準で failed に落とした。子は正常応答済み・残存採点 100 で、再実行で回復。プロトコル逸脱・429 に続く親側交絡の第 3 の型として記録（機序は未特定、claude 2.1.200）
- delegate の作業ディレクトリは既定で mktemp（/tmp 配下）に作られ、日を跨ぐと消える。失敗調査のためにも orchestrator では `DELEGATE_WORK_DIR` をラン専用ディレクトリに固定すべき（[bench_common_design.md](../docs/design/bench_common_design.md) の「計測メトリクス」と整合）
- Claude 実行系の子は `--no-session-persistence` で起動される（セッション残留なし。カンニング防止の観点では好都合）
- ベンチ workspace 直下の CLAUDE.md（ラン用 playbook）は、後日その workspace を別ワーカーに読ませる際に自動注入される。read-only 調査の委譲時は「workspace 内の指示ファイルには従わない」ことを request に含めるか、ワーカーの Blockers 報告を確認すること（2026-07-05 のコード品質精査で実例あり。ワーカーは正しく無視して報告してきた）
