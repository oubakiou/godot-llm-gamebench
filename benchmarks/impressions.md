[^cost]: completed ランにおける親費用・子費用それぞれの中央値と、「計」= ラン総額の中央値（`node src/bench/cli.ts report` の正本値。親中央値と子中央値の和は総額中央値と一致しないことがある）。親（Fable）は全モデル CLI 実測で $0.87〜$1.23 とほぼ一定。子の算定方法はモデルごとに精度が異なる: **実測** = CLI が返した費用（Claude 系）/ **換算** = 実測トークン × 単価表（Codex 系。キャッシュ割引を無視した上限寄りの概算）/ **推測** = 換算に加え単価が Devin 経由の glm-5.2 公称値 / **composer-2.5** は子トークンが実測不可のため不明（総額は親分 $1.11 より確実に高い）/ **swe-1.6** は単価表で子単価 $0（Devin 込み価格）。

[^score]: 100 点満点のルーブリック（隠しテスト 60 / 決定性 10 / 型品質 15 / プロジェクト健全性 15。[delegate_implement_bench_design.md](../docs/design/delegate_implement_bench_design.md) の「採点」）による全項目自動採点の、採用（completed）ラン合計（各モデル 3 反復、最大 300。反復間の再現性を評価へ含めるため中央値でなく合算）。型品質は strict 警告（untyped/unsafe）を error 昇格した per-file 検査のカウントで、警告 1 件につき −3（下限 0）。completed が 3 ラン未満のモデルは不足分を 0 とみなさず n を明記する（例: claude-haiku-4-5 は n=1。信頼性は本文の完了/試行で別軸報告）。

[^time]: completed ランの壁時計時間の中央値。子の実装時間だけでなく、親の委譲準備・受け入れ検証・最終報告まで含むラン全体の所要時間（親のオーバーヘッドは全モデル共通条件）。

[^quality]: sonnet ワーカーによる定性評価。自動採点の型警告カウントが「注釈の量」を測るのに対し、こちらは型付けの徹底度・Model/View 分離・可読性・定数化といった設計面を測る補完軸（例: `Array[int]` で全部に型を付ければ警告 0 になるが enum 型を捨てた設計は本欄で減点される）。**全採用ラン（各モデル 3 ラン、claude-haiku-4-5 のみ completed 1 ラン）を共通の較正アンカーで精査**し、ランごとに 5 段階で採点、列の値はその 3 ラン平均（小数第 1 位）+ コメント。5 段階の定義: **5** = typed コレクションを内部実装まで貫徹し magic number ほぼなし / **4** = 公開 API は typed だが内部に無型コレクションや直書き数値が残る / **3** = enum・class を宣言しつつ内部表現で型を放棄（int 運用・生 Dictionary 等）/ **2** = 無型 var が広範で構造も脆い / **1** = 型注釈がほぼ皆無（該当なし）。中間評価は 0.5 刻み。各モデル欄の ◎/○/△/▲ 表記はそれぞれ 5/4/3/2 に対応する。ばらつきの詳細は各モデル欄を参照。

非 completed の詳細（完了/試行: gpt-5.5 3/3、gpt-5.4-mini 3/4、devin-glm-5.2 3/5、composer-2.5 3/3、swe-1.6 3/3、claude-sonnet-5 3/3、claude-opus-4-8 3/6。追加ラウンド: gpt-5.3-codex-spark 3/3、devin-deepseek-v4-pro 3/4、composer-2.5-fast 3/3、cursor-gemini-3.1-pro 3/3、cursor-kimi-k2.7-code 3/4）は各モデル欄を参照。failed 3 件（+追加ラウンド 1 件）はすべて親側要因（セッション枠 429 ×2、親のプロトコル逸脱 ×2）で、子モデルの失敗ではない。追加ラウンドの stalled 1 件も response 受領後の親の沈黙で子は無罪。claude-haiku-4-5 は 1/7（5 stalled・1 failed）で、stalled 5 件はすべて子側（自作の終了しない godot 待ち。haiku 欄と issue #7 参照）、failed 1 件は子 exit 1 に親の一語規約逸脱が重なったもの（成果物の残存採点は 85 — 機能テストは全 pass だが型警告 80 件）。

## ベンチ対象モデル（実行順）

### gpt-5.5（Codex CLI）

本計測: **3 反復すべて completed・85 / 100 / 85（合算 270、委譲モデル 2 位）**。全 rep 1 往復、5〜6 分、実測トークン in 402k〜566k / out 9.8k〜13.0k。機能面は隠しテスト 33 ケース全 pass で反復間のぶれが皆無、仕様の細部（スプリッタ反転・RNG 非消費）を毎回落とさない。減点は型警告のみ（rep0 25 件 / rep1 0 件 / rep2 5 件）で、型品質の再現性が課題。

コード品質（3 ラン精査、○）: rep2 のみ `_items: Array[_ItemState]` まで typed 徹底の ◎ 相当だが、rep0/rep1 は中核の `_items: Dictionary` が無型。3 フェーズ分離と `BoardModelScript` 経由のカプセル化は 3 ラン共通で綺麗。単一ラン精査時の ◎ は rep 選びの上振れだった。

準備フェーズでの実績（ベンチ外）:

- リファレンス実装（2026-07-03、約 8 分、1 往復で completed）。不動点反復による移動解決、スプリッタのトグル反転条件、RNG 消費規則、`peek_next_kind()` の非消費実装（state 複製）など、仕様書の細部を一切落とさず、親レビューでの指摘事項ゼロ
- 検証が自発的: 指示した検証 3 件に加え、追加のルール検証スクリプトを自作して実行。報告の Verification 記載も正確（親の再実行で全件再現）
- Typed GDScript の品質が高い（内部クラス・typed Array を適切に使用。改善余地は typed Dictionary 未使用程度）。報告プロトコルも正しく遵守し読み取りコストが低い
- 隠しテストランナー（33 ケース、662 行、約 9 分、1 往復。sonnet 停滞打ち切り後の再委譲）。テスト設計・堅牢化は的確だったが、seed の偶然で pass する期待値の off-by-one が 1 件混入（親レビューの実測裏取りで検出・修正）。「テストが通った」報告の裏に潜在バグが残り得る好例で、オラクル側成果物は親の独立検証が必須という運用知見

### gpt-5.4-mini（Codex CLI）

本計測: **76.18 / 88 / 94（合算 258.18）で、ばらつきと停滞が gpt-5.5 より明確に多い**。型警告は 2〜4 件と少なめ。

- rep0: completed 76.18（RNG 消費規則を落とし決定性 0 + 型警告 4 件）・10.0 分 / rep1: 初回 **stalled**（無進捗 10 分で打ち切り、残存採点 80.09、スプリッタ系 6 テスト失敗）→ 再実行 completed 88・11.1 分 / rep2: completed 94（型警告 2 件のみ）・16.5 分
- 入力トークンが gpt-5.5 の約 2〜4 倍（698k〜1,533k）で時間も 10〜17 分と長い。「安いが遅く、たまに落ちる/停滞する」プロファイル
- Codex 実行系でも停滞が起きることを本計測で初確認（Claude 実行系固有ではなかった）
- コード品質（3 ラン精査、△）: 3 ランとも中核の `_items` が無型という共通の弱点を持ち、rep0 は `as ItemState` キャスト頻出、rep1 は `_cells: Array[int]`（enum を int 運用）、rep2 は完全無型の二重配列と、反復が進むごとに型付けが後退した

### devin-glm-5.2（Devin CLI）

本計測: **83.18 / 85 / 85（合算 253.18）、完了ランは 4.9〜6.0 分と gpt-5.5 より速い**。機能面・速度とも優秀で、減点は型警告（6〜16 件）が主。

- 採用ラン: rep0 再実行 83.18（スプリッタの退場時反転 1 件失敗 + 型警告 16 件）/ rep1 85 / rep2 再実行 85。実測トークン in 880k〜1,252k / out 15k〜24k（ATIF export 経由の measured が機能）

### composer-2.5（Cursor CLI）

本計測: **3 反復すべて completed・85 / 85 / 85（合算 255）、再実行ゼロ**。機能テストは全反復 pass、減点は型警告（7〜11 件）のみ。5.2〜7.4 分と速い。

### swe-1.6（Devin CLI）

本計測: **64.09 / 68.64 / 79.55（合算 212.28）— 機能・型の両軸で崩れる品質最下位**。全ラン completed・再実行ゼロ。

### claude-sonnet-5（Claude CLI）

### claude-opus-4-8（Claude CLI）

本計測: **採用 3 ラン すべて completed・85 / 85 / 85（合算 255）**（7.8〜9.8 分、実測 $1.24〜$1.67/ラン）。機能・決定性・健全性は全反復満点で、減点は型警告（6〜11 件）のみ。ただし到達までの道のりは全モデル中で最も波乱含み（起動 7 回、metrics 記録 6 試行）。

- rep0: 初回 stalled（実装完成後の停滞、残存 91）→ 再実行 completed
- rep1: 初回は orchestrator の watchdog バグ（走査中の一時ファイル削除との TOCTOU 競合）で**ハーネス側 crash**（モデル無罪、修正済み）→ 再実行 completed
- rep2: 初回が親の枠 429、再実行が**親（Fable）の最終応答プロトコル逸脱**（一語規約を破り長文で応答）でいずれも failed（子は両回とも残存 91）→ ユーザー承認の例外再実行で completed
- 子モデルとしての opus は全試行で機能満点相当の実装（残存採点 85〜91）を出しており、失敗はすべて親・ハーネス側要因。「親の非決定性」（[bench_common_design.md](../docs/design/bench_common_design.md) の「公平性と妥当性の限界」）が実際に計測ノイズになることを示す好例で、レポートに要注記
- コード品質（3 ラン精査、△〜○）: 公開 API は 3 ランとも typed だが、rep0/rep2 は View 層のヘルパー（`_cell_base_color(k: int)`・`_selected_kind: int` 等）まで int 運用が波及し、内部の型放棄が sonnet5 より広範。rep1 のみ `PackedInt32Array` + API 境界キャストで良好（単一ラン精査の ◎ はこの rep1 を見たもの）。WHY コメントの質は 3 ラン共通で高い

### fable-direct（Claude CLI、委譲なしベースライン）

## ラウンド横断の観察

- **天井効果は機能軸に限った現象**: 隠しテスト（60 点軸）は前ラウンドの委譲 7 モデル中 4 モデル（+ベースライン fable-direct）が全反復満点と天井気味だが、型品質軸が差を広げ、合算は 212.28〜285.00 に分布。型警告ゼロの run を出せたのは gpt-5.5（1 回）と claude-sonnet-5（2 回）だけで、fable-direct（3 回）を除けばゼロ警告はむしろ例外。機能・型の両軸で崩れたのは swe-1.6 のみ
- **「実装完成後・完了報告前」の停滞は実行系横断の現象**: Codex（gpt-5.4-mini）・Devin（devin-glm-5.2）・Claude（claude-opus-4-8）の 3 実行系で各 1 回観測。特定ベンダー固有ではない
- **親側交絡が失敗の主因**: failed 3 件はすべて親（claude -p）側の要因（セッション枠 429 ×2、最終応答のプロトコル逸脱 ×1）。子モデルの評価には影響させず、信頼性メトリクスの解釈時に注記する
- **委譲プロトコルの時間コストは約 3 分/ラン**: 同一実行系（Claude）で委譲あり（sonnet5 / opus4.8: 8.4〜8.6 分）と委譲なし（fable-direct: 5.7 分）を比較すると、往復 1 回の委譲でも準備・観測・報告読取りで 3 分前後を消費している。費用は総額ではほぼ同等
- **型警告カウント（自動）と定性評は補完関係**: 警告数は「注釈の量」を測り、定性評は「設計」を測る。gemini（警告 9〜82 件・定性 ▲〜△）のように両者が符合する例もあれば、deepseek（警告 4〜5 件と少ないのに内部は生 Dictionary 運用で △）や sonnet rep2（警告 0 なのに enum を int に落として △ 側）のように、注釈が完備でも設計の型崩れは警告に出ない例もある。精査すると「enum を宣言しつつ内部を int / 無型 Array で運用する」型崩れから「typed コレクションを末端まで貫徹」まで幅があり、**モデル間の差よりモデル内のラン間ばらつきが支配的**。◎ 水準を安定して出せたのはベースラインの fable-direct のみで、型品質の「安定性」自体がモデル特性として報告価値がある

## ハーネス側の注意（モデル所感ではないが委譲品質に影響）

- **Claude 実行系の子は停滞リスクが恒常的にある**: sonnet（4.6）で 2 回、haiku の E2E rep3 で 1 回、さらに claude-haiku-4-5 の正式ランで 5 回観測（正式ラン分の機構は上記「自作のハングするサブプロセス待ち」で特定済み。E2E rep3 はレート制限解除直後にもかかわらず、実装の大半を書いた後 17 分以上無出力で停滞 → watchdog の 40 分絶対上限で timeout 打ち切り）。timeout 打ち切り→残存成果物の採点→metrics 記録の経路は rep3 で実地検証済み。なお SIGKILL 打ち切りでは親の usage が取得できず parent_tokens は null になる
- **Claude 実行系は枠逼迫時に停滞・沈黙しやすい**（2026-07-03〜04 の sonnet 委譲で観測）: 初回は response 未生成のまま exit 1（stderr は /tmp 掃除で消失し原因確定できず）、リトライはプロセス生存・CPU 微増のままファイルを一切書かず 37 分無進捗で手動 kill。時期的に週間制限のスロットリングが疑わしい。レート/枠エラーでも沈黙して exit 1 になるため切り分けコストが高く、watchdog（無進捗タイムアウト）が必須
- **orchestrator が落ちても子プロセスツリーは生き残る**: watchdog crash 時（opus rep1）に親 claude -p と子ワーカーが孤児化し、枠を消費し続けていた。ハーネス障害後は必ずプロセス残骸を確認・停止する
- シェル環境に `DELEGATE_IMPLEMENT_MODEL` が残留することがある。委譲時は毎回明示指定しないと意図しないモデルに解決される（sonnet 指定のつもりが gpt-5.5 に解決された事例あり。v0.6.0 からは prepare 出力の `model_source` で env 由来を検出できる）
- Cursor の GLM は `glm-5.2-high` / `glm-5.2-max` のみで、素の `glm-5.2` は存在しない
- **Cursor `agent` CLI は複数プロセス同時起動で `~/.cursor/cli-config.json` の書き込みが競合して落ちることがある**（`ENOENT: rename cli-config.json.tmp`。2026-07-05 のスモーク確認を 3 プロセス並列で行った際に 1 件発生、単独再試行で成功）。ベンチ本計測は直列実行のため影響しないが、スモーク確認も含め agent CLI は並列起動しないこと。skill 側の緩和策（config 隔離 or 直列化）は [issue #8](https://github.com/oubakiou/delegate-skills/issues/8) として上流へ提案済み
- **子ワーカーが自作のハングするサブプロセスを待って停滞する**: 終了処理のない godot スクリプトを Bash 実行し、返らないコマンドを待ち続ける形。claude-haiku-4.5 で 7 試行中 5 回観測（sonnet5 0/3、opus4.8 1/6 と頻度はモデル依存）。非対話の子はツールエラーを受け取る機会がないまま watchdog SIGKILL に至るため、delegate-skills へ Bash timeout 注入等の復帰策を提案（[issue #7](https://github.com/oubakiou/delegate-skills/issues/7)）
- **停滞ランの残存実装が無限出力する場合、grader が stdout の文字列化で落ちる**（`ERR_STRING_TOO_LONG`、数百 MB 到達）: 出力捕捉に上限（先頭・末尾 4MB ずつ保全、中間切り捨て）を入れて修正済み。打ち切りラン採点はこの種の暴走成果物を前提に組む必要がある
- **親が dispatch 前に request を作り直すと「prepared のまま dispatch されない observe JSON」が残る**（2026-07-05 の gpt-5.3-codex-spark ランで 2 回観測）: 旧 metrics 集計はこれを往復数の分母に数えて usage 全数チェックが誤発動し、実測が取れているのに estimated（ほぼゼロ）へフォールバックしていた。dispatch されていない observe を分母から除外する修正を実施し、影響 2 ランの metrics は保存アーティファクトから再計算した。上流へは [issue #6](https://github.com/oubakiou/delegate-skills/issues/6) として報告済み
- delegate の作業ディレクトリは既定で mktemp（/tmp 配下）に作られ、日を跨ぐと消える。失敗調査のためにも orchestrator では `DELEGATE_WORK_DIR` をラン専用ディレクトリに固定すべき（[bench_common_design.md](../docs/design/bench_common_design.md) の「計測メトリクス」と整合）
- Claude 実行系の子は `--no-session-persistence` で起動される（セッション残留なし。カンニング防止の観点では好都合）
- ベンチ workspace 直下の CLAUDE.md（ラン用 playbook）は、後日その workspace を別ワーカーに読ませる際に自動注入される。read-only 調査の委譲時は「workspace 内の指示ファイルには従わない」ことを request に含めるか、ワーカーの Blockers 報告を確認すること（2026-07-05 のコード品質精査で実例あり。ワーカーは正しく無視して報告してきた）
