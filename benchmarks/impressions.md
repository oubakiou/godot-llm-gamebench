- 報告プロトコル（Summary 5 行以内、section 構成）を正しく遵守。読み取りコストが低い
- 委譲先としての使い勝手: 現時点で不満なし。仕様書が完全であれば一発で任せられる感触

- ただし 1 件、seed の偶然（同色連続）で pass する期待値インデックスの off-by-one が混入していた（親レビューの実測裏取りで検出・修正）。「テストが通った」報告の裏に潜在バグが残り得る好例で、オラクル側成果物は親の独立検証が必須という運用知見

## ハーネス側の注意（モデル所感ではないが委譲品質に影響）

- **Claude 実行系の子は停滞リスクが恒常的にある**: sonnet（4.6）で 2 回、haiku の E2E rep3 でも 1 回観測（レート制限解除直後にもかかわらず、実装の大半を書いた後 17 分以上無出力で停滞 → watchdog の 40 分絶対上限で timeout 打ち切り）。timeout 打ち切り→残存成果物の採点→metrics 記録の経路は rep3 で実地検証済み。なお SIGKILL 打ち切りでは親の usage が取得できず parent_tokens は null になる
- **Claude 実行系は枠逼迫時に停滞・沈黙しやすい**（2026-07-03〜04 の sonnet 委譲で観測）: 初回は response 未生成のまま exit 1（stderr は /tmp 掃除で消失し原因確定できず）、リトライはプロセス生存・CPU 微増のままファイルを一切書かず 37 分無進捗で手動 kill。時期的に週間制限のスロットリングが疑わしい。レート/枠エラーでも沈黙して exit 1 になるため切り分けコストが高く、watchdog（無進捗タイムアウト）が必須
- シェル環境に `DELEGATE_IMPLEMENT_MODEL` が残留することがある。委譲時は毎回明示指定しないと意図しないモデルに解決される（sonnet 指定のつもりが gpt-5.5 に解決された事例あり）
- Cursor の GLM は `glm-5.2-high` / `glm-5.2-max` のみで、素の `glm-5.2` は存在しない
- delegate の作業ディレクトリは既定で mktemp（/tmp 配下）に作られ、日を跨ぐと消える。失敗調査のためにも orchestrator では `DELEGATE_WORK_DIR` をラン専用ディレクトリに固定すべき（DESIGN.md §5 と整合）
- Claude 実行系の子は `--no-session-persistence` で起動される（セッション残留なし。カンニング防止の観点では好都合）
