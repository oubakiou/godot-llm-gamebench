> ベンチ準備・運用中の delegate-implement 委譲で得た、親エージェント（Fable）視点の主観的観察。サンプル数は小さく、定量結果の正本（`node src/bench/cli.ts report` の集計）とは区別して扱う。委譲のたびに追記する。

† completed ランにおける親費用・子費用それぞれの中央値（中央値同士のため合計は総額中央値と一致しない）。親（Fable）は全モデル CLI 実測で $0.87〜$1.23 とほぼ一定。子の算定方法はモデルごとに精度が異なる: **実測** = CLI が返した費用（Claude 系）/ **換算** = 実測トークン × 単価表（Codex 系。キャッシュ割引を無視した上限寄りの概算）/ **推測** = 換算に加え単価が Devin 経由の glm-5.2 公称値 / **composer-2.5** は子トークンが実測不可のため不明（総額は親分 $1.11 より確実に高い）/ **swe-1.6** は単価表で子単価 $0（Devin 込み価格）。

「自動テストによる評価」は 100 点満点のルーブリック（隠しテスト 60 / 決定性 10 / 型品質 15 / プロジェクト健全性 15。DESIGN.md §6）による全項目自動採点で、3 反復（completed ラン）の中央値。

§ completed ランの壁時計時間の中央値。子の実装時間だけでなく、親の委譲準備・受け入れ検証・最終報告まで含むラン全体の所要時間（親のオーバーヘッドは全モデル共通条件）。

‡ 自動採点の型警告カウント（全モデル 15/15）では差が出ないため、各モデルの中央値ランの workspace を sonnet ワーカーが精査した定性評価（◎優 / ○良 / △可 / ▲やや難）。観点は型付けの徹底度・Model/View 分離・可読性。相対順位: gpt-5.5 ≈ claude-sonnet-5 > claude-opus-4-8 > composer-2.5 ≈ gpt-5.4-mini > devin-glm-5.2 > swe-1.6。根拠は各モデル欄を参照。

非 completed の詳細（完了/試行: gpt-5.5 3/3、gpt-5.4-mini 3/4、devin-glm-5.2 3/5、composer-2.5 3/3、swe-1.6 3/3、claude-sonnet-5 3/3、claude-opus-4-8 3/6）は各モデル欄を参照。failed 3 件はすべて親側要因（セッション枠 429 ×2、親のプロトコル逸脱 ×1）で、子モデルの失敗ではない。

## ベンチ対象モデル（実行順）

### gpt-5.5（Codex CLI）

本計測: **3 反復すべて completed・100/100 満点**。全 rep 1 往復、5〜6 分、実測トークン in 402k〜566k / out 9.8k〜13.0k。隠しテスト 33 ケース全 pass で反復間のぶれが皆無。仕様の細部（スプリッタ反転・RNG 非消費）を毎回落とさない再現性が突出している。

コード品質（rep1 精査、◎）: `Array[CellKind]` の flat 配列を `get_cell` まで一貫して型付けし、退場→移動→詰まりの 3 フェーズ分離が明確。定数化が徹底し magic number がほぼ無い。

準備フェーズでの実績（ベンチ外）:

- リファレンス実装（2026-07-03、約 8 分、1 往復で completed）。不動点反復による移動解決、スプリッタのトグル反転条件、RNG 消費規則、`peek_next_kind()` の非消費実装（state 複製）など、仕様書の細部を一切落とさず、親レビューでの指摘事項ゼロ
- 検証が自発的: 指示した検証 3 件に加え、追加のルール検証スクリプトを自作して実行。報告の Verification 記載も正確（親の再実行で全件再現）
- Typed GDScript の品質が高い（内部クラス・typed Array を適切に使用。改善余地は typed Dictionary 未使用程度）。報告プロトコルも正しく遵守し読み取りコストが低い
- 隠しテストランナー（33 ケース、662 行、約 9 分、1 往復。sonnet 停滞打ち切り後の再委譲）。テスト設計・堅牢化は的確だったが、seed の偶然で pass する期待値の off-by-one が 1 件混入（親レビューの実測裏取りで検出・修正）。「テストが通った」報告の裏に潜在バグが残り得る好例で、オラクル側成果物は親の独立検証が必須という運用知見

### gpt-5.4-mini（Codex CLI）

本計測: **中央値 100/100 だが、ばらつきと停滞が gpt-5.5 より明確に多い**。

- Codex 実行系でも停滞が起きることを本計測で初確認（Claude 実行系固有ではなかった）
- コード品質（rep2 精査、○）: 命名は良いが `_cells` などが無型 2 次元配列（Variant 頼み）で型がやや緩い。Timer + signal は正攻法

### devin-glm-5.2（Devin CLI）

本計測: **中央値 100/100、完了ランは 4.9〜6.0 分と gpt-5.5 より速い**。品質の上限・速度とも優秀。

- 採用ラン: rep0 再実行 98.18（スプリッタの退場時反転 1 件のみ失敗）/ rep1 100 / rep2 再実行 100。実測トークン in 880k〜1,252k / out 15k〜24k（ATIF export 経由の measured が機能）

- 「完了報告の手前で止まる」停滞 1 回が唯一の減点材料
- コード品質（rep1 精査、△）: enum を宣言しつつ内部表現を一貫して `int` にダウングレード（`kind: int` / `_get_cell(): int`）しており型システムの意義が薄い。3 段階の処理構造自体は良い

### composer-2.5（Cursor CLI）

本計測: **3 反復すべて completed・100/100 満点、再実行ゼロ**。gpt-5.5 と並ぶ完全記録で、5.2〜7.4 分と速い。

- 全ラン 1 往復。停滞・プロトコル違反が一度もなく、委譲先としての安定感は今ラウンド最良クラス

### swe-1.6（Devin CLI）

本計測: **中央値 83.64 — 今ラウンドで唯一、品質で明確に差が付いたモデル**。79.09 / 83.64 / 94.55 で全ラン completed・再実行ゼロ。

- rep1 は入力 3.16M トークンと突出しており、試行錯誤の多さが伺える
- 運用面は安定（停滞・プロトコル違反なし）。「速く終わるが品質保証は薄い」プロファイルで、composer-2.5 と対照的

### claude-opus-4-8（Claude CLI）

本計測: **採用 3 ラン すべて 100/100 満点**（7.8〜9.8 分、実測 $1.24〜$1.67/ラン）。ただし到達までの道のりは全モデル中で最も波乱含み（起動 7 回、metrics 記録 6 試行）。

- rep0: 初回 stalled（実装完成後の停滞、残存 100）→ 再実行 completed
- rep1: 初回は orchestrator の watchdog バグ（走査中の一時ファイル削除との TOCTOU 競合）で**ハーネス側 crash**（モデル無罪、修正済み）→ 再実行 completed
- rep2: 初回が親の枠 429、再実行が**親（Fable）の最終応答プロトコル逸脱**（一語規約を破り長文で応答）でいずれも failed（子は両回とも残存 100）→ ユーザー承認の例外再実行で completed
- 子モデルとしての opus は全試行で満点相当の実装を出しており、失敗はすべて親・ハーネス側要因。「親の非決定性」（DESIGN §7）が実際に計測ノイズになることを示す好例で、レポートに要注記
- コード品質（rep1 精査、◎）: `PackedInt32Array` 採用 + 境界のみ `as CellKind` キャストという効率重視の型設計。フェーズごとの同時処理を説明する WHY コメントが秀逸。`(c + 1) as CellKind` の enum 宣言順依存が唯一の軽微な脆さ

### fable-direct（Claude CLI、委譲なしベースライン）

本計測: **3 反復すべて completed・100/100 満点、再実行ゼロ**。5.1〜6.6 分・$2.33〜$2.94/ラン（委譲なしのため全額親側、実測）。

- 位置づけ: 他 7 モデルと異なり**委譲プロトコルを通らない別条件**。親（claude-fable-5）が direct playbook で自ら実装・自ら検証する。「委譲のオーバーヘッド」と「品質上限」の参照点として追加（2026-07-05、ユーザー指示）
- **委譲オーバーヘッドの実測値が得られた**: 同じ Claude 実行系の委譲ラン（sonnet5 8.6 分 / opus4.8 8.4 分、総額 $2.27〜$2.35）に対し、直接実装は 5.7 分・$2.62 と**約 3 分速く費用ほぼ同等**。委譲プロトコル（リクエスト生成・子起動・観測・報告読取り）の時間コストが正味で見える
- コード品質（rep2 精査、◎）: `Dictionary[Vector2i, bool]` と Dictionary の型引数まで明記するのは今ラウンド唯一。`StepResult` の配列も全型付き、定数化徹底で直書きなし。唯一の差分候補は signal を使わず `is_finished()` を毎 tick polling する点
- 解釈上の注意: 実装者（fable）はこのベンチの運用者（親）と同一モデル。コード品質の精査は別モデル（sonnet）が行ったが、自己評価的バイアスの余地は注記しておく

## ラウンド横断の観察

- **天井効果が確定**: 7 モデル中 6 モデルが品質中央値 100。品質で差が付いたのは swe-1.6 のみで、モデル間の差は主に効率と信頼性（速度・コスト・停滞率）に現れた
- **「実装完成後・完了報告前」の停滞は実行系横断の現象**: Codex（gpt-5.4-mini）・Devin（devin-glm-5.2）・Claude（claude-opus-4-8）の 3 実行系で各 1 回観測。特定ベンダー固有ではない
- **親側交絡が失敗の主因**: failed 3 件はすべて親（claude -p）側の要因（セッション枠 429 ×2、最終応答のプロトコル逸脱 ×1）。子モデルの評価には影響させず、信頼性メトリクスの解釈時に注記する
- **委譲プロトコルの時間コストは約 3 分/ラン**: 同一実行系（Claude）で委譲あり（sonnet5 / opus4.8: 8.4〜8.6 分）と委譲なし（fable-direct: 5.7 分）を比較すると、往復 1 回の委譲でも準備・観測・報告読取りで 3 分前後を消費している。費用は総額ではほぼ同等
- **自動採点で差が出ない領域はコード品質の定性評で差が付いた**: 型警告カウントは全モデル 15/15 だが、精査すると「enum を宣言しつつ内部を int / 無型 Array で運用する」型崩れ（devin-glm-5.2、swe-1.6）から「typed Array を末端まで貫徹」（gpt-5.5、Claude 系）まで幅がある。型付けの徹底度は隠しテストの失敗傾向（swe-1.6）とも相関して見えた

## ハーネス側の注意（モデル所感ではないが委譲品質に影響）

- **Claude 実行系の子は停滞リスクが恒常的にある**: sonnet（4.6）で 2 回、haiku の E2E rep3 でも 1 回観測（レート制限解除直後にもかかわらず、実装の大半を書いた後 17 分以上無出力で停滞 → watchdog の 40 分絶対上限で timeout 打ち切り）。timeout 打ち切り→残存成果物の採点→metrics 記録の経路は rep3 で実地検証済み。なお SIGKILL 打ち切りでは親の usage が取得できず parent_tokens は null になる
- **Claude 実行系は枠逼迫時に停滞・沈黙しやすい**（2026-07-03〜04 の sonnet 委譲で観測）: 初回は response 未生成のまま exit 1（stderr は /tmp 掃除で消失し原因確定できず）、リトライはプロセス生存・CPU 微増のままファイルを一切書かず 37 分無進捗で手動 kill。時期的に週間制限のスロットリングが疑わしい。レート/枠エラーでも沈黙して exit 1 になるため切り分けコストが高く、watchdog（無進捗タイムアウト）が必須
- **orchestrator が落ちても子プロセスツリーは生き残る**: watchdog crash 時（opus rep1）に親 claude -p と子ワーカーが孤児化し、枠を消費し続けていた。ハーネス障害後は必ずプロセス残骸を確認・停止する
- シェル環境に `DELEGATE_IMPLEMENT_MODEL` が残留することがある。委譲時は毎回明示指定しないと意図しないモデルに解決される（sonnet 指定のつもりが gpt-5.5 に解決された事例あり。v0.6.0 からは prepare 出力の `model_source` で env 由来を検出できる）
- Cursor の GLM は `glm-5.2-high` / `glm-5.2-max` のみで、素の `glm-5.2` は存在しない
