## 1. 背景と方針

{現状のコードベース規模、責務集中、重複、テストしにくさ、DESIGN.md との乖離点を記述する。}

方針:

- **挙動不変なファイル移動を先**にする
- **public API / package exports / README の契約は変えない**。変える必要がある場合は feature plan として切り出す
- **DESIGN.md と乖離する変更を入れる場合は同じ PR で DESIGN.md を更新**する
- in-source test は実装と同じ移動先に追従させ、`npm run check` / `npm run test` を通す

## 2. 項目テンプレート

各候補は次の構造で記述する。

```markdown
### H?. <短いタイトル>

**対象**: <file:line>（複数の場合は箇条書き可）

**現状**: <責務混在 / 重複 / 結合度などの問題説明>

**分割案**:

- `<新ファイル / 新モジュール パス>`: <そこに置く責務>
- ...

**効果**: <保守性 / 拡張性 / テスタビリティの向上>

**リスク**: 低 / 中 / 高 + 根拠
```

すでに実装済み、または不採用と判断した候補は本文を消さず、状態を明示して残す。

```markdown
### M?. (完了済み | 不採用) <短いタイトル>

**状態**: **完了済み** / **不採用** — <理由 / commit / 既存実装の位置>
```

## 7. 共通の進め方

- 各候補は **1 PR = 1 候補** を原則とする
- 「ファイル分割のみ」と「責務再配置を含む構造変更」が混ざる場合は 2 PR に分ける
- in-source test は実装と同じ移動先に追従させる
- 実装後は `npm run check` / `npm run test` を通す
- 挙動不変を狙う候補では、サブエージェントで独立レビューし、等価性・テストカバレッジ・循環参照・公開 API 変更の有無を確認する
- DESIGN.md と乖離する変更は DESIGN.md 更新を同 PR に含める
- 計画書に含まれる候補がすべて完了または不要になったら、`docs/archive/<topic>.archive.md` に移動する
